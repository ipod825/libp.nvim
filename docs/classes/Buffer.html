Buffer:edit (opts)
Enters "edit mode" for the Buffer. "Edit mode" is a unified way of doing buffer content "renaming". For e.g., a file explorer plugin can use edit mode to implement file rename functionality. A git plugin can use edit mode to implement branch renaming, etc.

Parameters:

  • opts
    • get_items function A function that returns arbitrary result. This function is called two times: when users enters edit mode and after users save the buffer (:w), leaving edit mode. The two results are then passed to the update function for the plugin author to implement their own logic.
    • update function(ori_items,new_items) A function that is called with the two outputs of get_items after the users leave edit mode with :w.
    • fill_lines function function A function that is called when users just enter edit mode. Plugin authors could use this function to modify the buffer content to giver users a better editing experience.

Buffer:get_or_new (opts)
Returns a new Buffer or an existing one. Note that although this is defined as a member function, the caller should be the Buffer class or classes inheriting buffer. The return type is the caller class. The main difference between this function and open_or_new is that it does not open the Buffer in any window. The caller can then decide how to open the Buffer later (even in a floating window).

Parameters:

  • opts
    • filename string The file name of the corresponding vim buffer.

Returns:

    Buffer The Buffer object.

See also:

    Buffer:open_or_new

Usage:

    local InheritedBuffer = require("libp.ui.Buffer"):EXTEND()
local existing_or_new = InheritedBuffer:get_or_new({ filename = "filename" })
assert(existing_or_new:IS(InheritedBuffer))
Buffer:init (opts)
Constructor.

Parameters:

  • opts
    • filename string The file name of the corresponding vim buffer. (default nil)
    • content {string} or function The content to be filled in the corresponding vim buffer. Could be direct content of type array of string or a function that returns a string, which is passed as cmd to Job:init. (default {})
    • content_map_fn function({string})->string The function that maps each line of the content before it's filled in the vim buffer. Should only be useful when content is from Job. (default identity)
    • content_highlight_fn function(buffer,beg,ends,lines,context)->table

      The function that returns highlight info for the content. The signature of content_highlight_fn is:

      • buffer: The buffer under operation.
      • beg: The start line number, see :help api-indexing.
      • ends: The ending line number, see :help api-indexing.
      • context: Arbitrary table for the callee to store internal states (per Buffer).
      • return: a table with keys of names as vim.api.nvim_buf_add_highlight's arguments'.
      (default nop)
    • job_on_stdout_buffer_size number Passed to Job:init's onstdoutbuffer_size for optimizing buffer loading speed. (default 5000)
    • buffe_enter_reload boolean Whether to invoke reload on BufEnter. (default nil)
    • mappings table The table defining mappings in all modes. For e.g.

        mappings = {
      n = { j = function() end },
      v = { k = function() end },
  }

      Note that the functions are invoked inside in plenary.async.void such that you could define async action for key mappings. Also note that if a Buffer is also constructed inside a plenary.async.void context. The user might trigger the mapping before your Buffer's internal states are fully initialized and hit error. In such case, use set_mappings to set the mappings after initialization is done. (default nil)

    • b table buffer variables to be set. See :help vim.b. (default {})
    • bo table buffer options to be set. See :help vim.bo. (default {})
